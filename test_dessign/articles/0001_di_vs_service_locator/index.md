依存管理のアプローチが異なる。

本記事では DI とサービスロケーターの特徴を自分なりに整理し、比較していきたいと思います。

## それぞれの特徴

DI もサービスロケーターも依存を管理するためのデザインパターンです。<br>
どちらも基本的にはアプリケーションのエントリーポイントで使用するクラスや interface に対して
解決する具象クラスを決定し、それらの依存を使用する箇所で、指定したオブジェクトを使用可能にするパターンです。

最大の違いは依存の取得方法です。<br>
それぞれの依存解決方法を見ていきます。

### DI(Dependency Injection)

DI は依存しているモノ(クラス、interface)を取得する際に、外部から依存を注入してもらうことにより、依存するモノを取得します。<br>

具体的にはコンストラクタ、メソッド、プロパティを経由し依存を外部から注入(Injection)してもらいます。

```php
// DI で管理される依存
interface UserRepositoryInterface
{
    public function findById(int $id): User;
    public function save(User $user): void;
}

class UserController
{
    public function changeEmail(
        // 依存しているモノをメソッドの引数に注入してもらう
        UserRepositoryInterface $userRepository,
        int $userId,
        string $newEmail): void
    {
        $user = $userRepository->findById($userId);
        $user->email = $newEmail;
        $userRepository->save($user);
    }
}

```
### サービスロケーター

一方、サービスロケーターはそのクラスが依存しているモノを自発的に取得しにいきます。

多くは、依存管理用のライブラリが提供している依存取得メソッド(下記例では `resolve`)などを使用し、取得します。

```php
// こちらも同様に UserRepositoryInterface が定義されている。

class UserController
{
    public function changeEmail(
        int $userId,
        string $newEmail): void
    {
        // resolve 関数を使用し自分がほしい依存を取得する
        $userRepository = resolve(UserRepositoryInterface::class);
        $user = $userRepository->findById($userId);
        $user->email = $newEmail;
        $userRepository->save($user);
    }
}
```

## サービスロケーターのデメリット

ここからが一般的にサービスロケーターがアンチパターンと言われる理由になります。

### 依存が外から見えない

依存しているクラスやインタフェースを引数で明示している DI に対して、サービスロケーターは依存しているクラス内で依存を取得しているため、外部から依存が隠れます。

そのため、そのクラスが何に依存しているのか内部のコードを読まないとわかりません。
これがさらにそのクラスが依存しているクラスも何に依存しているかわからないということになり、依存の隠蔽が連鎖します。

一方で、依存のバインド箇所を一部に限定すれば、サービスロケーターで管理しているクラスが何に解決されるかはわかるので、依存箇所を把握する必要がない気もします。また、結局のところ引数経由で依存するか、内部で `resolve` するかの違いであり、Language Server などの力を借りればどちらも依存箇所を一覧で取得できるため大差ない気もします。

しかし、これをユニットテストなど書き出すと途端に扱いづらくなります。
なぜなら、ユニットテストする際にテスト対象のクラスが何に依存しているかをすべて把握する必要があり、
それらをテスト実行前に適切にバインドする必要があります。

つまり、テスト対象のクラスが依存しているもの、さらに依存しているものが依存しているものをすべて把握したうえでないとユニットテストができなくなります。

DI を利用した場合もテスト対象の依存を解決する必要がありますが、こちらは引数で管理されるため、コンパイラによる保護で把握できます。

### 依存管理のライブラリに依存する

DI もサービスロケーターも元の使用動機はクラスやライブラリへの依存を柔軟に管理することですが、
サービスロケーターは依存取得のため `resolve` などの専用ライブラリの機能に依存しなければならないという矛盾が生じます。

ユニットテストを書く際なども、DI の場合は引数に渡すオブジェクトをモックなどに差し替えるだけですむため、
依存管理のライブラリを必要としませんが、サービスロケーターの場合は、ユニットテストですらもライブラリに依存せざるを得なくなります。

筆者は普段、PHP の Web フレームワークである Laravel をよく使いますが、Laravel ではサービスコンテナという Laravel ネイティブ機能で依存管理を提供しています。<br>
サービスロケーターを使用した場合、ユニットテストでも Laravel フレームワークの機能に依存することとなり、
テスト実行の度にフレームワークのブート処理を実行します。
これがなかなか処理時間を必要とし、結果的に軽量に書きたいユニットテストですらも実行時間がかかるようになってしまいます。

また、フレームワークの機能に依存するということは、そのフレームワークの庇護の下でしか動かないということでもあります。
本来、技術的な関心とは切り離すべきドメイン/ビジネス層のクラスが、フレームワークなしでは動かないというのはおかしな話です。
そして、フレームワークが寿命を迎え、新しいプラットフォームに移植する際に、これらのクラスからフレームワークの依存を取り除くという、神経の使う手術が必要となります。

---

以上のように、通常のプロダクトコードを書くだけではさほど害はないサービスロケーターですが、
ユニットテストを書き出すと途端にやりづらさを感じます。

そして、サービスの寿命が延びた現代において、保守性、拡張性の向上の観点から自動テストの存在は必須であり、
その自動テストが書きづらいということは、多くのデメリットがあります。

また、テストの書きやすいクラスは設計も優れている傾向にあります。<br>
サービスロケーターを使用し、テストの書きづらいクラスは設計にも問題を抱えている可能性が高いです。

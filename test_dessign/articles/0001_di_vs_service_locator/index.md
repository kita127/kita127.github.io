## 本記事の目的

DI とサービスロケーターはどちらもアプリケーションの依存を管理するためのデザインパターン。
依存管理のアプローチが異なる。
現在ではサービスロケーターはアンチパターンとみなされ、DI の使用が推奨される。
本記事では DI とサービスロケーターの特徴を自分なりに整理し、比較していきたいと思います。

## それぞれの特徴

DI もサービスロケーターも依存を管理するためのデザインパターンです。<br>
どちらも基本的にはアプリケーションのエントリーポイントで使用するクラスや interface に対して
解決する具象クラスを決定し、それらの依存を使用する箇所で、指定したオブジェクトを使用可能にするパターンです。

最大の違いは依存の取得方法です。<br>
それぞれの依存解決方法を見ていきます。

### DI(Dependency Injection)

DI は依存しているモノ(クラス、interface)を取得する際に、外部から依存を注入してもらうことにより、依存するモノを取得します。<br>

具体的にはコンストラクタ、メソッド、プロパティを経由し依存を外部から注入(Injection)してもらいます。

```php
// DI で管理される依存
interface UserRepositoryInterface
{
    public function findById(int $id): User;
    public function save(User $user): void;
}

class UserController
{
    public function changeEmail(
        // 依存しているモノをメソッドの引数に注入してもらう
        UserRepositoryInterface $userRepository,
        int $userId,
        string $newEmail): void
    {
        $user = $userRepository->findById($userId);
        $user->email = $newEmail;
        $userRepository->save($user);
    }
}

```
### サービスロケーター

一方、サービスロケーターはそのクラスが依存しているモノを自発的に取得しにいきます。

多くは、依存管理用のライブラリが提供している依存取得メソッド(下記例では `resolve`)などを使用し、取得します。

```php
// こちらも同様に UserRepositoryInterface が定義されている。

class UserController
{
    public function changeEmail(
        int $userId,
        string $newEmail): void
    {
        // resolve 関数を使用し自分がほしい依存を取得する
        $userRepository = resolve(UserRepositoryInterface::class);
        $user = $userRepository->findById($userId);
        $user->email = $newEmail;
        $userRepository->save($user);
    }
}
```

## サービスロケーターはアンチパターン？

一般的にはサービスロケーターはアンチパターンと言われています。

### 依存関係が外から見えない

依存しているモノを引数で明示している DI に対して、サービスロケーターは依存しているクラス内で依存しているモノを取得しているため、外部から依存が隠れます。

そのため、そのクラスが何に依存しているのか内部のコードを読まないとわかりません。

### 依存管理のライブラリに依存する

DI もサービスロケーターも元の使用動機はクラスやライブラリへの依存を柔軟に管理することですが、
サービスロケーターは依存取得のため `resolve` などの専用ライブラリの機能に依存しなければならないという矛盾が生じます。

ユニットテストを書く際なども、DI の場合は引数に渡すオブジェクトをモックなどに差し替えるだけですむため、
依存管理のライブラリを必要としませんが、サービスロケーターの場合は、ユニットテストですらもライブラリに依存せざるを得なくなります。

筆者は普段、PHP の Web フレームワークである Laravel をよく使いますが、Laravel ではサービスコンテナという Laravel ネイティブの機能で依存管理を提供しています。<br>
サービスロケーターを使用した場合、ユニットテストでも Laravel フレームワークの機能に依存することとなり、
テスト実行の度にフレームワークのブート処理を実行します。
これがなかなか処理時間を必要とし、結果的に軽量に書きたいユニットテストですらも実行時間がかかるようになってしまいます。

DI vs サービスロケーター

本記事の目的

DI とサービスロケーターはどちらもアプリケーションの依存を管理するためのデザインパターンです。 違いとしては依存管理のアプローチが異なります。 現在ではサービスロケーターはアンチパターンとみなされ、DI の使用が推奨されています。 本記事では DI とサービスロケーターの特徴を整理し、サービスロケーターは本当にアンチパターンなのか、有用な使いどころはないのかを改めて考えたいと思います。

それぞれの特徴

どちらも基本的にはアプリケーションのエントリーポイントで使用するクラスや interface に対して 解決する具象クラスを決定し、それらの依存を使用する箇所で、バインドしたオブジェクトを使用するパターンです。

最大の違いは依存の取得方法です。
それぞれの依存解決方法を見ていきます。

DI(Dependency Injection)

DI は依存しているモノ(クラス、interface)を取得する際に、外部から依存を注入してもらうことにより、依存するモノを取得します。

具体的にはコンストラクタ、メソッド、プロパティを経由し依存を外部から注入(Injection)してもらいます。

// DI で管理される依存
interface UserRepositoryInterface
{
    public function findById(int $id): User;
    public function save(User $user): void;
}

class UserController
{
    public function changeEmail(
        // 依存しているモノをメソッドの引数に注入してもらう
        UserRepositoryInterface $userRepository,
        int $userId,
        string $newEmail): void
    {
        $user = $userRepository->findById($userId);
        $user->email = $newEmail;
        $userRepository->save($user);
    }
}

サービスロケーター

一方、サービスロケーターはそのクラスが依存しているモノを自発的に取得しにいきます。

多くは、依存管理用のライブラリが提供している依存取得メソッド(下記例では resolve)などを使用し、取得します。

// こちらも同様に UserRepositoryInterface が定義されている。

class UserController
{
    public function changeEmail(
        int $userId,
        string $newEmail): void
    {
        // 依存管理ライブラリが提供する resolve 関数を使用し自分がほしい依存を取得する
        $userRepository = resolve(UserRepositoryInterface::class);
        $user = $userRepository->findById($userId);
        $user->email = $newEmail;
        $userRepository->save($user);
    }
}

サービスロケーターはアンチパターン？

一般的にはサービスロケーターは以下の理由によりアンチパターンとみなされます。

  • 依存が外部から隠れる
  • 依存管理ライブラリに依存する

これらがなぜ問題があるかは後ほど説明します。

以上のような理由から、アンチパターンと言われるサービスロケーターですが、 この記事では一度、まっさらな気持ちで DI とサービスロケーターの特徴を比較し、なるべく先入観抜きで再評価をしてみようと思います。

DI とサービスロケーターの比較

依存の取得方法

  • DI: コンストラクタ/引数で明示される
  • サービスロケーター: 内部で resolve する

DI はコンストラクタやメソッドの引数で依存するクラスやインタフェースを明示し、そこに外部から依存を注入します。 一方、サービスロケーターは内部で resolve などを使用し、内部から直接依存を取得します.

依存しているクラスやインタフェースを引数で明示している DI に対して、サービスロケーターは依存しているクラス内で依存を取得しているため、外部から依存が隠れます。

そのため、そのクラスが何に依存しているのか内部のコードを読まないとわかりません。 これがさらにそのクラスが依存しているクラスも何に依存しているかわからないということになり、依存の隠蔽が連鎖します。

一方で、依存する実体を決めるバインド箇所を一部に限定すれば、依存管理しているクラスが何に解決されるかはわかるので、依存箇所を把握する必要がない気もします。また、結局のところ引数経由で依存するか、内部で resolve するかの違いであり、Language Server などの力を借りればどちらも依存箇所を一覧で取得できるため大差ない気もします。

しかし、ユニットテストになると途端に扱いやすさに差が出ます。 なぜなら、サービスロケーターの場合はユニットテストを作成する際にテスト対象のクラスが何に依存しているかをすべて把握する必要があり、それらをテスト実行前に適切にバインドする必要があります。

つまり、テスト対象のクラスが依存しているもの、さらに依存しているものが依存しているものをすべて把握したうえでないとユニットテストが作成できなくなります。

DI を利用した場合もテスト対象の依存を解決する必要がありますが、こちらは引数で管理されるため、コンパイラによる保護で把握できます。

引数の数

  • DI: 多くなることがある
  • サービスロケーター: 引数は減る

DI の場合は依存するクラスが多いと大量の引数がコンストラクタやメソッドにズラッと並ぶため、クラスが肥大化したように見えます。

サービスロケーターを使用すれば依存するクラスが resolve などでダイレクトに依存を取得できるため、この引数ズラッと問題も解消されます。

一見するとサービスロケーターのほうがスマートに見えますが、これには問題があります。

ひとつのクラスに依存する I/F の引数が大量に並ぶというのは、そもそも扱う関心が多すぎるのかもしれません。 つまりこれは単一責任原則に違反している可能性があります。

引数が多いということは、この原則に違反していることの示唆であり、設計を見直すシグナルとも言えます。

逆にサービスロケーターでは依存が多くても、引数にズラッと依存が並ばないためこの問題に気づきにくいです。

結局のところ、サービスロケーターを使用したところで依存している数には変わりはなく「臭いものに蓋をする」状況とも言えなくないです。

依存のバケツリレー

  • DI: ある
  • サービスロケーター: ない

例えば、システムのデータを扱うエンティティオブジェクトなどは、通常、依存管理ライブラリなどでバインドせず、new で生成しますが、これらに依存を渡そうとすると DI の場合はコントローラー層などのエントリポイントからバケツリレー方式で依存を渡していく必要があります。 DI で依存を注入する層からエンティティまでの階層が深い場合、依存を直接使用するのはエンティティにもかかわらず、それに渡すためだけに他のクラスはバケツリレーで依存を運搬する必要があります。

一方、サービスロケーターを使用した場合は、依存を使用したいエンティティが直接 resolve すればバケツリレーを回避し、ダイレクトに依存を取得できるようになります。

バケツリレーを回避し、依存を手軽に使用できるのは確かにサービスロケーターのメリットのひとつかと思います。

しかし、これについては別の問題が潜んでいる可能性があります。

そもそもこの深い階層構造の底で依存を利用する設計自体に問題があることを示唆しています。

まず、依存管理ライブラリで管理したい依存とは主にどんなものかというと、多くは外部に対して副作用のある I/F が大半ではないかと思います。 具体的には「メール送信」「ロギング」「データベースアクセス」「外部通信」「I/O」などです。 これらはレイヤで言えば技術的な関心を扱うインフラストラクチャー層の機能です。一方、エンティティなどのオブジェクトは複雑な事業の関心を扱うドメイン層のクラスです。 ドメイン層のクラスが技術的なクラスに依存していることがそもそも設計としては間違っている可能性があります。

これらの関心の分離ができていれば、ドメイン層はクリーンな状態となり、 依存管理ライブラリで管理する外部の I/F は比較的エントリポイントに近い浅い層だけで扱えるようになります。

そのため、このバケツリレー状態も設計改善のシグナルと言えます。

ライブラリへの依存

  • DI: ない
  • サービスロケーター: ある

DI もサービスロケーターも元の使用動機はクラスやライブラリへの依存を柔軟に管理することですが、 サービスロケーターは依存取得のため resolve などの専用ライブラリの機能に依存しなければならないという矛盾が生じます。

ユニットテストを書く際なども、DI の場合は引数に渡すオブジェクトをモックなどに差し替えるだけですむため、 依存管理のライブラリを必要としませんが、サービスロケーターの場合は、ユニットテストですらもライブラリに依存せざるを得なくなります。

筆者は普段、PHP の Web フレームワークである Laravel をよく使いますが、Laravel ではサービスコンテナという Laravel ネイティブ機能で依存管理を提供しています。
サービスロケーターを使用した場合、ユニットテストでも Laravel フレームワークの機能に依存することとなり、 テスト実行の度にフレームワークのブート処理を実行します。 これがなかなか処理時間を必要とし、結果的に軽量に書きたいユニットテストですらも実行時間がかかるようになってしまいます。

また、フレームワークの機能に依存するということは、そのフレームワークの庇護の下でしか動かないということでもあります。 ドメイン/ビジネス層のクラスが resolve を使用していた場合、フレームワークの機能に依存することになります。 本来、技術的な関心とは切り離すべきドメイン/ビジネス層のクラスが、フレームワークなしでは動かないというのはおかしな話です。 そして、フレームワークが寿命を迎え、新しいプラットフォームに移植する際に、これらのクラスからフレームワークの依存を取り除くという、神経の使う手術が必要となります。

比較結果

DI とサービスロケーターについて比較結果をまとめると以下の通りです。

| 比較項目 | DI | サービスロケーター | | --- | --- | --- | | 依存の取得方法 | コンストラクタ/引数で明示される | 内部で resolve する | | 引数の数 | 多くなることがある | 引数は減る | | 依存のバケツリリレー | ある | ない | | ライブラリへの依存 | ない | ある |

DI とサービスロケーターの特性を比較した場合、一見すると一長一短あるように見えますが、 DI のデメリットについては DI の問題というより、設計に根本的な問題がある可能性が高いと言えます。

また、サービスロケーターのデメリットはユニットテストを書く際に顕著に現れると考えます。 サービスの寿命が延びた現代においては保守性、拡張性を高めるプログラミングは必須であり、 そのためにはユニットテストなどの自動テストはもはや必須であるため、それらと相性のよい DI にやはり軍配があがるのではないでしょうか。

では、サービスロケーターの使いどころはないのでしょうか？

サービスロケーターの使いどころ

階層を深くせず、関心を適切に分離する。単一責任原則を守る。 これらは確かに理想ですが、理想通りいかないのが現実です。 パフォーマンスや歴史的経緯などを優先した場合に、設計を犠牲にしなければならないケースもあります。

また、クリーンな設計や丁寧なドメインモデルなどはエレガントですがコストもかかります。 アプリケーションの機能でもさほど複雑でもなく、競争優位性を産まない機能にこれらの手法を使うと、時にはオーバーエンジニアリングとなり、コストがメリットを上回ることもあります。

開発スピードを優先した場合、ドメイン層がインフラストラクチャ層に依存するレイヤードアーキテクチャを採用し、ORM でガリガリロジックを扱うアクティブレコードのほうが最適な場合もあります。

この場合は思い切ってサービスロケーターを採用し、ORM やロジック内でもガリガリ依存を扱ったほうが正解かもしれません。

しかし、この手法で開発したプログラムは長期の保守、拡張は難しく、それらを承知のうえで決断する必要があります。

結論

今回は一般的にはアンチパターンと言われるサービスロケーターについて、純粋な気持ちで DI と比較し、有用なケースがないか検討しました。

結論としては、自動テストやクリーンな設計を考慮した場合、一般的に言われるようにサービスロケーターはそれらとは相性が良くないため、極力避けるべきかと思います。

使いどころがあるとすれば、以下のようなケースでしょうか。

  • 高度な設計は不要
  • 保守性、拡張性が求められない機能
  • 比較的シンプルに実装できる機能
  • とにかく開発スピード重視

とは言え、DI を使用したからと言って格段、高度な設計が必要なわけでもなく、また、開発スピードにも大きな差はでないと思います。

結局のところ DI よりもサービスロケーターを積極的に使用する理由はほとんどないというのが結論でしょうか。

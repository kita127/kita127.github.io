## 単体テスト野郎

私は単体テスト野郎です。実装時にはほとんどの場合において TDD、テストファースを意識してプログラミングします。
Kent Beck の「テスト駆動開発」を読んでから数年間、このスタンスで仕事をしています。
単体テストを書くことは、それにかかるコストを上回るメリットがたくさんあると、実務の中でのたくさん経験してきました。

現在の職場は単体テストに対するリテラシーは比較的高く、全員がある程度、実装時にはテストを追加します。
そんな職場の中でも私のテストに対するこだわりは異質で、恐らく同僚は私の偏執を理解できないのではないかと思います。
それはテストを書く動機が私と同僚とでは異なっており、異質の原因はその動機の違いだと考えています。

同僚と話をする中で他の皆がテストを書く一番の動機はリグレッション(デグレード)に対する防衛のようです。
もちろん、私もリグレッション対策が単体テストを書く一つの動機ではあります。
単体テストを積み重ねることで、機能追加や拡張時のリグレッションを防いでくれることは明白な単体テストの利点です。

しかし、私はこの利点と同じかそれ以上に単体テストによる別の利点があると実感しています。

***テストを書くと圧倒的に設計が良くなる！！！！***

これが私が信じる単体テストにおける利点のもう一つの側面です。

これを同僚との話の中で話題に挙げるのですが、なかなか伝わりません。
かと言ってこの利点をしっかり説こうと思っても、一言で表せないためなかなか職場の雑談では伝えきれません。
いつも「テストを書くと設計も良くなりますよね！」とだけ言って、「何を言っているんだ、こいつは？」という間をいただきます。

今回はこのいつも一言で言い表せない想いを、このブログに綴っていこうと思います。伝われぇ！！

## 単体テストによりなぜ設計が良くなるのか？

### 前提 : テストファースト

それでは想いを綴っていきます。ですが、このテストによって設計が良くなる理論にはひとつ大前提があります。
それは必ず「TDD」あるいは「テストファースト」で実装を進めることです。

これはよくよく考えると当然で、後でテストを書くということは、すでにプロダクトコードが完成しているということなのでテストが設計の改善に影響しないからです。
一応、テストを書いてからリファクタするという選択肢もありますが、ここがまたテストファースとでやった場合と状況が異なってきます。
なぜなら後書きテストで実装した場合は、プロダクトコードもテストコード自体もテストファーストで取り組んだ場合よりも変更が難しくなります。
この理由も詳しく後ほど書きます。

以上により、今回語る「単体テストにより設計が良くなる」理論はプロダクトコードの実装よりも先にテストコードを書くことを前提に説明します。

## なんで良くなるの？ 理由 : テストを書きやすくしようとするから

これからあなたは開発しているシステムのある機能を実装していきます。テストファースで進めるため、まずはプロダクトコードの前にそのコードをテストするためのコードから書きます。

当然複雑なものは書きたくないため、テストのための準備(Arrange)も実行(Act)も検証(Asesrtion)もシンプルでわかりやすく書こうとします。
すると当然ながら、テスト対象のクラス(System under test)はテストをしやすいように作ろうという意識になります。

テストをしやすいクラスは設計の良いクラスになります。以上が良くなる理由です。

と言っても「なぜ良くなるの？」を説明していないため、ここをさらに深堀りしていきます。

### 理由その1 : 疎結合なクラスになる

テストをしやすくしようと思った場合、なるべくテスト対象クラスは独立して機能するようします。
テストするためにテスト対象クラス以外のクラスやモジュールがたくさん必要となると、当然テストするための準備(Arrange)が大変になります。
なるべく、テスト対象クラスが必要とする依存を減らした結果、自然、疎結合なクラスになります。
ご存知の通り、クラス設計は高凝集、疎結合が良いとされています。テストをシンプルに書こうとした結果、自然と結合度が優れたクラスになります。
